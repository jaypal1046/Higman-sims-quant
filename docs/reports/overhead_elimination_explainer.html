Why does the codebook exist at all?
The Higman-Sims graph has 100 vertices. Each vertex is a point in 22D space — a specific list of 22 numbers like [0.31, -0.72, 0.08, ..., 0.44].

The old approach: store those 22 numbers explicitly for every vertex so that at decode time you can look them up by ID. That's the codebook.

100 vertices × 22 dims × 32 bits = 70,400 bits overhead

But here's the insight you're pointing at: what if the receiver already knows the coordinates, without you sending them?
The codebook is overhead only when the receiver doesn't know it
If both sender and receiver share the same mathematical definition of the graph, the coordinates are implicit — they can be recomputed from the group structure alone. The codebook overhead drops to 0 bits.

The lookup cost is the same problem
Even if we eliminate storage overhead, we still need to find the nearest vertex fast. Traditional nearest-neighbour search is O(n). The symmetry group of HS (order 44,352,000) lets us reduce this dramatically — but we need to exploit it deliberately.
Solution A — shared spec
Both encoder and decoder know the HS/McLaughlin graph spec (open mathematics). Codebook never transmitted. Just store vertex IDs.
overhead: 0 bits · lookup: O(n)
Solution B — algebraic formula
Express vertex coordinates as closed-form algebraic expressions from the group generators. Reconstruct any vertex on-demand in O(1). No table at all.
overhead: 0 bits · lookup: O(1) ✓
Solution C — implicit coordinates best
Embed graph in a canonical basis derived from its adjacency eigenstructure. Every vertex ID maps to coordinates via a deterministic matrix multiply. Stateless. O(1).
overhead: 0 bits · decode: O(d) = O(22)
Solution D — compressed seed
Send a short random seed (64 bits). Both sides expand it into the full codebook via a shared PRNG. Amortized overhead near zero.
overhead: 64 bits · lookup: O(1) after init
How implicit coordinates work
The Higman-Sims graph has a known adjacency matrix A (100×100). Its eigenvectors form a canonical 22D embedding — this is proven by the graph's spectral theory. The embedding is uniquely determined by the graph structure itself.
A · V = V · Λ (eigendecomposition, done once at build time)
vertex_coords[i] = V[i, 0:22] (top 22 eigenvectors)

At encode: id = argmin_i ‖x − vertex_coords[i]‖
At decode: x̂ = vertex_coords[id] → O(22) multiply, no table
Why this is O(1) in practice
The 22D coordinate for vertex i is a fixed linear combination of eigenvectors. You can precompute a 100×22 matrix (just 17,600 floats — not transmitted, computed). Decoding vertex id is a single row lookup: one memory access = O(1).

The symmetry group of HS (automorphisms of order 44M) means the nearest-vertex search can be pruned dramatically: project your query vector onto the eigenspace, read off the nearest bin in O(log 100) = O(7) steps using the group orbits as a tree.
